Now add retention run history for Mougle AI cleanup operations.

Context:
The project already has:

1. ai_jobs table.
2. ai_job_events table.
3. ai_workers table.
4. AI retention cleanup service:
   - previewCleanup()
   - runCleanup()
5. Admin retention UI:
   - /admin/ai-retention
6. Worker identity hardening is complete.

Goal:
Persist every AI retention cleanup execution so admins can see who ran cleanup, when, with what policy, whether it was dry-run or real, and how many rows were eligible/deleted.

Do not rewrite the app.
Do not expose Python directly to the frontend.
Do not bypass the TypeScript API.
Do not add cron.
Do not auto-run cleanup.
Keep cleanup admin-controlled.

Part A — Database

Add a new table, for example:

ai_retention_runs

Suggested columns:

- id
- run_id
- admin_id nullable
- dry_run boolean not null
- policy JSON not null
- eligible_counts JSON not null
- deleted_counts JSON nullable
- status text not null
- error text nullable
- started_at timestamp not null
- completed_at timestamp nullable
- created_at timestamp not null

Suggested statuses:

- started
- succeeded
- failed

Indexes:
- created_at
- admin_id
- dry_run
- status

Use the existing database/ORM style. Do not introduce a new ORM.

Part B — Backend service

Update aiRetentionService.ts:

1. Add createRetentionRun(...)
2. Add completeRetentionRun(...)
3. Add failRetentionRun(...)
4. Add listRetentionRuns(...)

Behavior:

- Every POST /api/admin/ai-retention/run should create a retention run record.
- Dry-run executions should also be recorded.
- Real cleanup executions should record deleted counts.
- Failed cleanup attempts should record status=failed and a safe error message.
- Do not store secrets.
- Keep policy and count JSON compact.
- Use transactions where practical.

/admin/ai-retention

- run time
- admin id

- policy summary

UI behavior:
