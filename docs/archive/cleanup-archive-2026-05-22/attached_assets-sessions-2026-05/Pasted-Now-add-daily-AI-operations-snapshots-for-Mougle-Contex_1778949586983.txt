Goal:
Persist daily AI ops snapshots so admins can see trends over time:
- jobs processed per day
- failures per day
- worker health
- cleanup counts
- health status history

Do not rewrite the app.

Do not bypass the TypeScript API.
Do not add Redis.
Do not add cron yet.

Keep snapshots admin-controlled/manual for now.

Part A — Database

Add a new table, for example:

ai_ops_snapshots

Suggested columns:

- id
- snapshot_id
- snapshot_date
- generated_by_admin_id nullable
- health_status
- health_reasons JSON
- job_metrics JSON
- worker_metrics JSON
- retention_metrics JSON
- notification_metrics JSON nullable
- raw_summary JSON nullable
- created_at

Indexes:
- snapshot_date
- health_status
- generated_by_admin_id
- created_at

Rules:
- snapshot_date should be unique if practical, or enforce one snapshot per date unless force=true.
- Use existing database/ORM style.
- Do not introduce a new ORM.

Part B — Backend service

Add a service such as:

server/services/aiOpsSnapshotService.ts

Methods:

1. generateSnapshot({ adminId, date?, force? })

Behavior:
- call aiOpsSummaryService.getSummary()
- call aiOpsNotificationsService if useful
- create a compact snapshot row
- if a snapshot already exists for the date and force=false, return the existing snapshot
- if force=true, replace or update the existing snapshot safely
- do not store secrets or raw payloads
- keep JSON compact

2. listSnapshots({ since, until, healthStatus, limit, offset })

3. getSnapshot(snapshotId)

Part C — Backend routes

Add admin-only routes:

1. POST /api/admin/ai-ops/snapshots

Body:
{
  "date": "YYYY-MM-DD optional",
  "force": false
}

Response:
{
  "snapshot": {...},
  "created": true|false
}

2. GET /api/admin/ai-ops/snapshots

Filters:
- since
- until
- healthStatus
- limit
- offset

3. Optional:
GET /api/admin/ai-ops/snapshots/:snapshotId

Security:
- requireRootAdmin
- unauthenticated users cannot access
- normal users cannot access
- do not expose secrets
