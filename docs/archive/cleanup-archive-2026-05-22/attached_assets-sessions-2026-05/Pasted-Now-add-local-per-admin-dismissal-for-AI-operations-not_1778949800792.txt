Now add local per-admin dismissal for AI operations notification nudges.

Context:
The project already has:

1. Admin AI Ops Summary:
   - GET /api/admin/ai-ops/summary
   - /admin/ai-ops

2. Admin AI Ops Notifications:
   - GET /api/admin/ai-ops/notifications

3. Daily AI Ops Snapshots:
   - POST /api/admin/ai-ops/snapshots
   - GET /api/admin/ai-ops/snapshots
   - AI Ops Trends card on /admin/ai-ops

Goal:
Allow admins to dismiss AI ops notification nudges locally in the browser until the underlying condition changes.

Do not rewrite the app.
Do not add backend persistence for dismissals yet.
Do not add Redis.
Do not add cron.
Do not add email/SMS/push notifications.
Do not expose Python directly to the frontend.
Keep this local-only using localStorage.

Part A — Dismissal model

Each notification already has stable fields:

- id
- severity
- title
- message
- count
- actionLabel
- actionUrl
- createdAt
- metadata

Add a local dismissal key that changes when the condition changes.

Recommended condition fingerprint:

dismissKey = notification.id + ":" + notification.severity + ":" + notification.count + ":" + stable JSON of notification.metadata

Behavior:

1. If the same notification condition appears again with the same dismissKey, keep it dismissed.
2. If count, severity, or metadata changes, show it again.
3. Dismissals are per browser and per admin user where possible.
4. Store in localStorage.
5. Do not send dismissal data to the server.

Part B — Frontend utility / hook

Add a small frontend helper or hook, for example:

client/src/hooks/use-ai-ops-notification-dismissals.ts

It should support:

- getDismissedKeys()
- dismissNotification(notification)
- restoreNotification(notification)
- clearAllDismissals()
- isDismissed(notification)
- visibleNotifications
- dismissedNotifications

If there is an existing hook pattern, follow it.

Part C — UI changes on /admin/ai-ops

Update the “Attention Needed” panel:

1. Show only visible, non-dismissed nudges by default.
2. Add a “Dismiss” button to each nudge.
3. Add a collapsed “Dismissed” section if there are dismissed nudges.
4. Allow restoring a dismissed nudge.
5. Add “Clear dismissed” button.
6. If all nudges are dismissed, show:
   “All current AI ops nudges are dismissed.”
7. If there are no nudges at all, keep:
   “No AI ops issues detected.”

Part D — Safety / UX

1. Do not dismiss critical nudges forever if their condition changes.
2. Do not hide new critical nudges that have a different fingerprint.
3. Keep action links visible for dismissed nudges in the dismissed section.
4. Keep styling compact.
5. Keep manual refresh behavior working.
6. On refresh, recompute dismissal visibility from current notifications.
7. No backend changes required unless TypeScript types need small cleanup.

Run existing checks/tests.

- dismiss hides a nudge.
- dismissed nudge appears in the dismissed section.
- restore brings it back.

- changed condition fingerprint would reappear.
