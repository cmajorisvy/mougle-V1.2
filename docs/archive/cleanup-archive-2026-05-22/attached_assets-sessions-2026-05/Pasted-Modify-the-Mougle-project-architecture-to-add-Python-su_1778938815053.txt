1. User-facing AI agents
2. Local in-house Mougle AI agents for advanced complex work

Do NOT convert the existing core app away from Node.js / TypeScript.

Important:
Do NOT convert the React frontend, Vite app, Express API, admin dashboard, auth, billing, routes, render workflow, Replit deployment setup, GitHub workflow, or existing TypeScript services to Python.

Keep Mougle core app as:

React / Vite frontend
↓
TypeScript / Express API
↓
Agent Orchestrator / Job Queue
↓
Python AI Worker Layer
↓
Postgres / Vector DB / Object Storage

Goal:
Add a separate Python AI-agent layer that supports:

A. User AI agents
These are agents available to Mougle users for complex tasks such as:
- content research
- newsroom assistance
- claim extraction
- fact-checking support
- document analysis
- semantic clustering
- summarization
- media analysis
- audio/video processing
- personalized AI workflows
- long-running user jobs
- batch processing
- report generation

These are internal Mougle agents used by the platform/admin team for advanced operations such as:
- quality evaluation
- LLM output scoring
- moderation assistance
- duplicate detection

- source credibility analysis
- vector search experiments
- model comparison
- agent benchmarking
- system monitoring intelligence
- research notebooks
- internal workflow automation
- advanced ML experiments

16. Add a README explaining the architecture clearly.

Design rules:

- User-facing agents and in-house agents must be clearly separated.
- User agents must never access admin-only or internal-only workflows by default.
- In-house agents must be marked as internal/admin-only.
- All jobs should go through a controlled job router.
- TypeScript should create jobs and control permissions.
- Python should consume jobs, execute AI/ML work, and return results.
- Long-running complex work should be handled by Python workers, not the TypeScript request/response path.
- Do not expose Python agents directly to the frontend.
- The frontend should only call the TypeScript API.
- The TypeScript API should decide whether a job is allowed, then pass the job to Python.

Preferred long-term pattern:

User or admin action
↓
React frontend
↓
TypeScript / Express API
↓
Permission checks
↓
Job creation
↓
Python job router
↓
Correct user or in-house AI agent
↓
Result saved to database/storage
↓
TypeScript API returns result/status to frontend
