Now add AI job history and monitoring views for Mougle.

Context:
The project already has:

1. Postgres-backed ai_jobs table.
2. TypeScript AI job service.
3. Python worker bridge.
4. Real Python handlers.
5. User-facing UI:
   - post claim extraction panel
   - /ai-tools semantic clustering page
6. Admin-facing UI:
   - /admin/newsroom-package

Goal:
Add persistent AI job history views so users and admins can track past and current AI jobs.

Do not rewrite the app.
Do not expose Python directly to the frontend.
Do not bypass the TypeScript API.
Do not add Redis.
Do not require external AI APIs.
Keep Python worker optional.

Part A — Backend API

1. Add a user-facing endpoint:

GET /api/ai-jobs

This should return the authenticated user’s AI jobs only.

Support filters:
- status
- jobType
- origin=user only for normal users
- limit
- offset or cursor
- date range if easy

Return fields:
- jobId
- jobType
- origin
- status
- createdAt
- updatedAt
- completedAt
- failedAt
- source summary if available
- compact result summary if available
- statusUrl

Security:
- require authentication
- users must not see other users’ jobs
- users must not see inhouse jobs

2. Add an admin endpoint:

GET /api/admin/ai-jobs

This should return all AI jobs for root/admin users.

Support filters:
- status
- jobType
- origin
- requestedByUserId
- requestedByAdminId
- lockedBy
- limit
- offset or cursor
- date range if easy

Return fields:
- jobId
- jobType
- origin
- status
- requestedByUserId
- requestedByAdminId
- lockedBy
- retryCount
- maxRetries
- createdAt
- updatedAt
- lockedAt
- startedAt
- completedAt
- failedAt
- source summary
- error summary

Security:
- require root/admin authorization
- do not expose secrets
- do not expose raw internal error stack traces

3. Reuse the existing aiJobService where practical.
   Add methods such as:
   - listUserJobs(...)
   - listAdminJobs(...)
   - summarizeJob(...)
   - summarizeResult(...)

4. Keep GET /api/ai-jobs/:jobId behavior unchanged.
