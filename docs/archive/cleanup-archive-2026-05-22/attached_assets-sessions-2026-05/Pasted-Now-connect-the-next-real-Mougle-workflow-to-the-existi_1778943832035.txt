   - POST /api/ai-jobs/claim-extraction

Goal:
Add a real TypeScript/API workflow that allows authorized Mougle users or admins to enqueue semantic clustering for multiple posts or text items and later retrieve the persisted result.

Do not rewrite the app.
Do not expose Python directly to the frontend.
Do not bypass the TypeScript API.
Do not connect Python directly to Postgres.
Do not add Redis.
Do not require paid external AI APIs.
Keep Python optional.

Implementation requirements:

1. Add or extend an endpoint for semantic clustering.

Suggested endpoint options:

POST /api/ai-jobs/semantic-clustering

and/or, if there is a project/newsroom/post collection route:

POST /api/posts/semantic-clustering
POST /api/newsroom/semantic-clustering
POST /api/admin/newsroom/semantic-clustering

Use the project’s existing route conventions.

2. The endpoint should support clustering real content from existing posts where possible.

Recommended request body shape:

{
  "postIds": ["..."],
  "items": [
    {
      "id": "optional-client-id",
      "text": "...",
      "title": "optional title"
    }
  ],
  "title": "optional clustering job title"
}

3. Support both modes if practical:
   - postIds: load existing post content from the database
   - items: direct text items supplied by the authenticated user

4. Validate input:
   - require at least 2 items total
   - reject empty text
   - limit number of items, for example max 100
   - limit text size per item, for example max 25,000 chars
   - trim whitespace
   - return 400 for invalid input

5. Permission checks:
   - require authentication
   - users may only cluster posts they are allowed to access
   - admins may cluster broader newsroom/internal content only through admin routes
   - normal users must not access in-house/admin-only jobs

6. Enqueue a vector.clustering job using aiJobService.

7. The job payload should include enough information for the Python clustering handler:
   - item id
   - text
   - title if available
   - source type, such as post or direct_text
   - source reference where available

8. Return 202 with:
   - jobId
   - status
   - statusUrl
   - source summary
   - item count

9. Use the existing AI job status endpoint where possible:
   GET /api/ai-jobs/:jobId

10. Do not allow users to read other users’ clustering job results.

11. Keep admin/internal access separate.

12. Do not modify Python handler logic unless the existing payload shape requires a small compatibility fix.

Validation:

- Run Python smoke test if Python files are touched.
- Confirm authenticated semantic clustering job creation works.
- Confirm unauthenticated request returns 401/403.
